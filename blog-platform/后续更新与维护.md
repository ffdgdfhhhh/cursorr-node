# 创享社区：后续更新与维护指南

本文面向**已完成首次部署**的环境，说明日常发版、运维检查与备份习惯。首次安装与 Nginx 细节请参考同目录下的 `部署教程.md`。

---

## 一、更新前确认清单

- [ ] 本地已自测通过（登录、发帖、聊天室等关键路径）。
- [ ] 若改了 `server/package.json`，记住上线后要在服务器执行 `npm install --omit=dev`。
- [ ] 若改了数据库结构，确认 Sequelize / 迁移策略，并**先备份数据库**再更新。
- [ ] 准备好**不覆盖**服务器上已有的 `server/.env`（或使用 Git 时确保 `.env` 在 `.gitignore` 中）。

---

## 二、日常发版流程（推荐顺序）

### 2.1 仅前端变更（Vue / 静态资源）

1. 本地构建：

   ```bash
   cd client
   npm install   # 依赖有变更时
   npm run build
   ```

2. 将 **`client/dist`** 整目录上传到服务器，覆盖站点下的同名目录（路径需与你当前 Nginx root 或 Node 静态目录一致）。

3. **若整站由 Node 托管静态**：建议执行一次 `pm2 restart chuangxiang`，避免偶发缓存或进程状态异常。

4. 浏览器用 **强制刷新**（Ctrl+F5）或**无痕窗口**验证，排除旧 JS 缓存。

### 2.2 仅后端变更（API、Socket、业务逻辑）

1. 上传变更后的 **`server`** 目录文件（勿上传本地 `node_modules`；勿覆盖 `.env`）。

2. SSH 进入服务器：

   ```bash
   cd /www/wwwroot/你的域名/blog-platform/server
   npm install --omit=dev
   pm2 restart chuangxiang
   ```

3. 若修改了 `.env`：

   ```bash
   pm2 restart chuangxiang --update-env
   ```

### 2.3 前后端同时变更

合并执行 **2.1** 与 **2.2**：先上传 `dist` 与 `server`，再在 `server` 目录安装依赖并 **PM2 重启**。

## 三、用 Git 简化更新（可选）

**永远不要**把含真实密码的 `.env` 提交到 Git。

### 在别处首次克隆整个项目

```bash
git clone https://github.com/ffdgdfhhhh/cursorr-node.git
cd cursorr-node/blog-platform
```

克隆后需在 `server/`、`client/` 分别执行 `npm install`，并自行创建 `server/.env`（详见仓库根目录 `README.md` 与本文第一节）。

### 以后更新（拉取远程最新代码）

```bash
cd cursorr-node
git pull
```

随后进入 **`blog-platform/`** 按下方流程安装依赖、构建（如需）并重启服务。

### 服务器或本机已有仓库时：拉取后部署

进入你机器上的 **`blog-platform`** 目录（若整库克隆则为 `…/cursorr-node/blog-platform`；若曾直接放在站点根下则可能为 `/www/wwwroot/你的域名/blog-platform`，以实际为准）：

```bash
cd /www/wwwroot/你的域名/blog-platform
git pull
cd server && npm install --omit=dev && cd ..
# 若在服务器上构建前端：
cd client && npm install && npm run build && cd ..
pm2 restart chuangxiang
```

若前端仅在本地构建，则 `git pull` 后只同步本地上传的 `client/dist` 即可。

## 四、健康检查与日志

| 目的 | 命令或操作 |
|------|------------|
| 后端是否存活 | `curl -s http://127.0.0.1:3000/api/health`（应返回 JSON 且 `success` 为 true） |
| 查看实时日志 | `pm2 logs chuangxiang` |
| 查看进程状态 | `pm2 status` |
| Nginx 改配置后 | 宝塔中保存并重载 Nginx，或 `nginx -t` 通过后 `systemctl reload nginx`（以实际环境为准） |

出现 **502** 时：先确认 `curl 127.0.0.1:3000/api/health` 正常，再查 Nginx 反代与 PM2。

## 五、数据库与用户文件备份

- **MySQL**：在宝塔「计划任务」中设置定期**数据库备份**，或手动导出 `.sql`。
- **用户上传**：若使用 `server/uploads`（或等价目录），备份时应与数据库策略一致（与代码版本分开存放）。
- 恢复前先在测试环境验证，避免直接覆盖生产且无回滚。

---

## 六、安全与配置维护

- 定期更换数据库密码时：同步修改 `server/.env` 中的 `DB_*`，并 `pm2 restart chuangxiang --update-env`。
- **`JWT_SECRET`** 更换会导致已登录用户全部失效，需评估停机窗口。
- 保持 Node 为 LTS 版本（如 18+）；大版本升级前在副本环境验证。
- 不要将 `.env`、数据库备份放到公开网盘或公开仓库。

## 七、常见问题速查

| 现象 | 建议处理 |
|------|----------|
| 页面仍是旧版 | 确认 `dist` 已覆盖；浏览器强刷 / 无痕；若用 CDN 需刷新缓存 |
| 控制台仍请求旧 IP 的 API | 检查 `client/.env.production` 是否使用相对路径（见部署教程）；重新 build 并上传 `dist` |
| `Cannot find module` | 在 `server` 目录执行 `npm install --omit=dev` |
| WebSocket / 聊天不可用 | 检查 Nginx 是否开启 WebSocket，参考 `deploy/nginx.example.conf` |
| SPA 路由 404 | Nginx `location /` 需 `try_files $uri $uri/ /index.html;` |

更完整的首次部署与架构说明见 **`部署教程.md`**。

## 八、建议的发布节奏（可选）

- 小改动：按第二节流程直接发版。
- 大改动：预留回滚（保留上一份 `dist` 压缩包与上一版 `server` 备份或 Git tag）。
- 变更记录：在仓库或团队文档中简单记下日期与内容，便于线上问题追溯。

文档对应仓库路径：`blog-platform/`。
